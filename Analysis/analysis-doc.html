1Introduction

Pong Tracker provides Users with a centralized website for recording and viewing statistics of Beer Pong games. Details of each game are entered into the website as Users are playing in real time. This analysis document details how the Pong Tracker product works. It provides information on models, user interfaces, and the associations between them.

2Purpose and Uses

The purpose of Pong Tracker is to provide a service to Users that is simple, intuitive and attractive. Pong Tracker allows Users to engage in friendly competition, without the complexity of managing the system themselves. Pong Tracker provides:

  • Users
    • Ability to create, score and confirm a Game
    • Ability to view personal statistics, and those of other Users
    • Ability to manage their own account.
    • Ability to view rules
  • Administrators
    • Ability to login and logout
    • Ability to manage Institutions
    • Ability to ban and unban Users

3Object Classification

3.1Entity Objects

Game A Game being recorded using Pong Tracker. A Game has two Teams.
Cup A Cup used in a Game. Each Team has six Cups.
Team A Team participating in a Game. A Team has two Users.
LifeStats The cumulative statistics collected and stored for a User over time.
Ranking The statistics involved with TrueSkill.
Event An individual incident that occurs during a Game. These are added to LifeStats upon Game confirmation.
EventType The description of an Event.
Institution The institution or school a User is enrolled in.
Administrator The actor who has special permissions within Pong Tracker. Administrators are not involved in Games.
User The actor who interacts with Pong Tracker.

3.2Boundary Objects

CreateGameView Where a User creates a new Game for tracking.
ScoreGameView Where a User scores a Game by selecting a cup, an action, and the applicable User.
EndGameView Where a User selects the winning Team, and under what circumstances the win was obtained.
ConfirmGameView Where a User approves or denies the validity of the Game.
StatisticsView Where a User sees their own or other Users' statistics and general profile information.
SearchUserView Where a User can search for other Users.
LeaderBoardView Where a User can see the top ranked Users.
RulesView Where a User can read the rules of Pong Tracker and Beer Pong.
RegistrationView Where a User registers with Pong Tracker.
EditProfileView Where a User edits their profile on Pong Tracker.
UserLoginView Where a User logs into their Pong Tracker account.
UserLogoutView Where a User logs out of their Pong Tracker account.
DeleteAccountView Where a User may delete their account from Pong Tracker.
AddInstitutionView Where an Administrator adds an Institution.
DeleteInstitutionView Where an Administrator deletes an Institution.
BanUserView Where an Administrator bans a User.
UnbanUserView Where an Administrator removes a ban from a User.
AdministratorLoginView Where an Administrator logs into their Pong Tracker administrative account.
AdministratorLogoutView Where an Administrator logs out of their Pong Tracker administrative account.

3.3Control Objects

CreateGameController Coordinates creating a Game.
ScoreGameController Coordinates scoring a Game.
EndGameController Coordinates the actions involved with ending a Game.
ConfirmGameController Coordinates confirming validity of a Game.
StatisticsController Coordinates retrieving and recording Stats about a User.
SearchUserController Retrieves User data on other Users. Used for looking at other User's statistics.
LeaderBoardController Retrieves User data to display on LeaderBoard View.
RegistrationController Coordinates adding a User.
EditProfileController Coordinates updating User information.
UserLoginController Retrieves User data to verify identity.
UserLogoutController Terminates User session on Pong Tracker.
DeleteAccountController Coordinates deleting a User.
AddInstitutionController Coordinates the actions involved when an Administrator adds an Institution.
DeleteInstitutionController Coordinates the actions involved when an Administrator deletes an Institution.
BanUserController Coordinates the actions involved when an Administrator bans a User.
UnbanUserController Coordinates the actions involved when an Administrator removes a ban from a User.
AdministratorLoginController Retrieves Administrator data to verify identity.
AdministratorLogoutController Terminates Administrator session on Pong Tracker.

4.3Use Case Class Diagrams

4.3.1User Class Diagrams

4.3.1.1Create Game Class Diagram

The CreateGameView has four textboxes (txtPlayerName1-txtPlayerName4). A User enters the name of all Users participating in the Game in the textboxes. The create button (btnCreate) is selected when all fields have been entered. There is an option to go back (btnBack) if the User no longer wants to create a Game.

4.3.1.2Score Game Class Diagram

The ScoreGameView has 12 buttons, each representing a Cup (btnTeam1Cup1-btnTeam2Cup6). When a Cup is sunk, a User selects the matching Cup button on the form. A User then selects how the cup was sunk which is either by a trick shot (btnTrickShot), by a bounce shot (btnBounceShot), by a party foul (btnPartyFoul), or by a simply sinking the ball (btnCupSunk). Lastly, a User selects which player sunk the Cup (btnPlayer1-btnPlayer4). A User may end the Game at any point by selecting abort (btnAbort) which cancels the Game without recording the information, or by selecting the end game button (btnEndGame) which brings the User to the EndGameView.

4.3.1.3End Game Class Diagram

The EndGameView has two sets of radio buttons. A User must select the winning team (rbtnWinner), and how the win was obtained (rbtnCause). Once the radio buttons have been selected, a User hits a confirm button (btnConfirm) to complete the form.

4.3.1.4Confirm Game Class Diagram

The ConfirmGameView has two buttons. There is a button to confirm (btnConfirm) or deny (btnDeny) the validity of the game. A User only selects one button. Once one button has been selected, the form is complete.

4.3.1.5View Statistics Class Diagram

The StatisticsView shows statistics User, along with their general profile information (username, first name, last name, etc.). There are no form elements on this page.

4.3.1.6View Other Users' Statistics Class Diagram

The SearchUserView has a textbox (txtSearchUser) where a User enters their search criteria. Clicking the search button (btnSearch) executes the search.

4.3.1.7View Leader Board Class Diagram

The LeaderBoardView has a dynamic drop down list for filtering the list.

4.3.1.8View Rules Class Diagram

The RulesView is a static html page with no form elements. It has Beer Pong and Pong Tracker website rules.

4.3.1.9Registration Class Diagram

The RegistrationView has a textbox where a User enters their email address (txtEmail) and a textbox to enter the username (txtUsername) they want. Once they have completed these fields, the select the sign up button (btnSignUp) to complete the form.

4.3.1.10Edit Profile Class Diagram

The EditProfileView has textboxes for username (txtUsername), password (txtPassword), first name (txtFirstName), last name (txtLastName), email (txtEmail), height (txtHeight) which is measured in centimeters, and year of graduation (txtYearOfGraduation). A User updates their information in the textboxes as necessary. A User can upload a picture (txtUserProfilePhoto) to their profile as well. To do this, a User hits the browse button (btnBrowse) to obtain an image from their computer. Next, a User hits upload (btnUpload) and the image is added to their profile. In order to save their updated information, a User hits save (btnSave).

4.3.1.11User Login Class Diagram

The UserLoginView has a textbox where a User enters their username (txtUsername), and a textbox where the User enters their password (txtPassword). Upon completion of these fields, a User selects the submit button (btnSubmit) in order to complete their login request.

4.3.1.12User Logout Class Diagram

The LogoutView has a button to logout (btnLogout). Selecting this button, terminates a User's session.

4.3.1.13Delete Account Class Diagram

The DeleteAccountView has a textbox (txtPassword) where a User must enter their password. Once this is done, a User selects the button called delete (btnDeleteAccount). This causes the account to be deleted.

4.3.2Administrator Class Diagrams

4.3.2.1Add Institution Class Diagram

The AddInstitutionView has a textbox (txtInstitutionName) where an Administrator enters the institution and an add button (btnAdd) to execute the addition request.

4.3.2.2Delete Institution Class Diagram

The DeleteInstitutionView has a combo box (cbInstitutionName) where an Administrator selects the existing institution they wish to delete and a delete button (btnDelete) to execute the deletion request.

4.3.2.3Ban User Class Diagram

The BanUserView has a textbox (txtUsername) where an Administrator enters the User they wish to ban, and a ban button (btnBan) to execute the request.

4.3.2.4Unban User Class Diagram

The UnbanUserView has a textbox (txtUsername) where an Administrator enters the User they wish to lift a ban from, and an unban button (btnUnban) to execute the request.

4.3.2.5Admin Login Class Diagram

The AdministratorLoginView has a textbox where an Administrator enters their username (txtUsername), and a textbox where the Administrator enters their password (txtPassword). Upon completion of these fields, an Administrator selects the submit button (btnSubmit) in order to complete their login request.

4.3.2.6Admin Logout Class Diagram

The AdministratorLogoutView has a button to logout (btnLogout). Selecting this button, terminates an Administrator's session.
