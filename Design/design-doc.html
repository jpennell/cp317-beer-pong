1Introduction

1.1Document Overview

This is the Design document for Pong Tracker, an online website which allows users to track their Beer Pong playing skills and compare them to others. This is part of the project documentation for CP317 during the Spring 2013 semester.

The first part is an overview of the governing guidelines for this project. The second part is about the logical design involving the architecture of Pong Tracker and how various elements interact. The contents of each package is described as is the database which the project will use. The third part covers the design of methods to be implemented. The fourth part will cover Django, our selecected web framework for implementation. The fifth part will act as an introduction to front end prototyping concluding with the final parts being references and SQA Version History.

1.2Target Audience

This document is for the implementation and front end teams as it contains detailed information regarding the implementation procedures.

1.3Guidelines

1.3.1Technical Guidelines

Pong Tracker is hosted on a Sun Microsystem Linux Server with a MySQL database. The Linux server must be able to serve HTML 5 and CSS 3. Intepretation of Python 2.7 is required. The server is also required to integrate with the Django web framework and Git. Django is a strong web framework with many tools built in such as sessions, and database APIs; Django will be discussed in detail later. The only concern is the implemention is using Python 2.7 when the current version is 3.3.2. 2.7 has been selected because Django support for 3.3.2 is currently only an experimental version.

Coding Guidelines and Conventions:

  • Must meet HTML 5 and CSS 3 standards
  • Server side code must be done using Python 2.7
  • Client Side code must be done using jQuery, JavaScript, and Ajax
  • Must be object-oriented
  • Code shall follow PEP-8 standards as outlined bellow.
    • 1 Tab for indents
    • Maximum Line length of 72 characters
    • Blank lines shall seperate logical paragraphs
    • All models shall have a uni code method
    • Imports exlcluding froms shall be on seperate lines
    • Do not use extraneous white space, however one space on both sides of mathematical symbols is required
    • Compound statements are discouraged
    • Commenting should kept up to date and accurate
    • There shall be no commented out code in the final
    • Use inlind comments sparingly
    • constants shall be upercase with underscore
    • '_'represents private
    • Capital letters on classes
  • Exceptions to PEP-8:
    • camel case method and variable names with a lower case first letter
    • Packages shall be capitalized
  • Note: There are built in django variables in methods that will follow PEP-8

  • PEP 8 shall also be used for documentation
    • """Description
    Keyword arguments:
    paramater -- desctiption (type)
    Contributors:
    Contributor names
    Output:
    return value (type)
    """

Testing:

  • All functions are unit tested.
  • Acceptance testing by the client is necessary.

1.3.2Ethical & Legal Guidelines

  • The application will make use of libraries created by third-party developers; however, to respect intellectual property rights, the application will be developed using libraries under open source licenses.
  • All coding must be accompanied by comments.

4Django Web Framework

4.1Framework Description & Functionality

Django is a strong web framework with many tools built in such as session handling, caching, authentication, and database APIs. Django focuses on taking a lot of the work away from the developer allowing for easier, more rapid development.

One word sums up how Django works, Magic. In more specific terms Django works by processing a given URL against the registered URLs and calling the respective view. That view will then use the required models potentially updating the database returning a HTTPs response. This response is usually a template (HTML page) which can then be displayed for the user. The majority of these templates will have buttons that trigger posts to URLs that will start the process all over again.

4.2How Django is Used

In order to use Django, the libraries are first imported for a developer to run to create the project. With a project a developer can create apps; the equivalent of packages. These packages contain the models, views, forms, templates, URLs, and tests for their respective package. Once the project is started a developer can start to hook up the models, templates, and views in a model view controller fashion.

We are currently using the Django libraries for administration, authentication, sessions, and static files. The student group on hopper does not have permissions to create databases as such the testing was rewritten in order to perform unit testing.

In the django.contrib package are many tools available to developers we have selected several:

    auth

     Django's built in authentication framework. This system handles the varios parts associated with a user: accounts, permissions, groups, and cookie-based user session. It provides password hashing, and basic user forms. It does not however provide password strength checking, throttling or login attempts. When the database is synced the required tables are generated taking another step away from the developer. This package is dependent upon the contenttypes app and session, and authentication middleware.

    sessions

     Django's session framework allows a developer to store and retrieve values in the session. Each user has one session which is stored on the server side, abtrasting the sending and recieving of cookies. Sessions are dependent upon the SessionMiddleware midlware that is included with django. While django does over cookie session support we have have selected the default of database-bakced sesssion.

    messages

     Django allows developers to send one time messages to a user by using cookie or session based messaging. The frame work allows for the temporary storage of messages in one request and the retrieval of it in another. The message can be retrieved by templates in the html to display to the user. Each message has a tag declaring its priority level. Messaging is dependent upon SessionMiddleware, MessageMiddleware, and django.contrib.messages.context_processors.messages as a template context processor.

    staticfiles

     Staticfiles uses finders to gather the static files associated with the project into a single location. Staticfiles is used by defining the listing the location of static files in the STATICFILES_DIRS variable of settings.py. We are using this framework to incorporate our css files into the project.
